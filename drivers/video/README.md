# Видеодрайверы смены режима для KolibriOS

Документация для разработчиков новых драйверов. Описывает **формат сервиса vidmode**
(простой, для нативных драйверов конкретных карт), **формат сервиса KMS/«DISPLAY»**
(богатый, для портов DRM-драйверов), мост **kms.sys** между ними и — главное — **когда
какой формат выбирать**.

> TL;DR: для конкретной старой карты пиши **нативный `.sys` в формате vidmode** (~сотни
> строк FASM, переиспользуешь ядро). Полноценный **KMS-стек** оправдан только для
> современных GPU. Если KMS-драйвер уже есть — не переписывай его под vidmode, а сделай
> тонкий **мост** (как `kms.sys`).

---

## 1. Зачем такая архитектура

Сменить видеорежим в KolibriOS раньше можно было только на загрузочном «синем экране».
Приложение **vidmode** (`programs/system/vidmode/`) делает это в рантайме: оно опрашивает
все драйверы, собирает их режимы, даёт выбрать драйвер+режим и применить (Apply).

Ключевые принципы:

- **Детект — в драйвере, не в приложении.** vidmode chip-agnostic: он не знает ничего о
  железе, просто грузит каждый драйвер по имени и спрашивает у него режимы. Новый драйвер
  **не требует правок приложения** (кроме одной строки в списке имён, см. §3.6).
- **Драйверы взаимозаменяемы.** Единый IOCTL-протокол из 3 команд — приложение одинаково
  работает с любым.
- **Переиспользование ядра вместо дублирования.** Драйвер программирует регистры чипа и
  зовёт готовую функцию ядра `SetLfbMode` — не дублирует блиттеры/конвейер вывода. VBE
  через `V86`-монитор ядра — не дублирует видео-BIOS.
- **Загрузка ≠ смена режима.** Загрузка драйвера (опрос) железо не трогает; режим меняется
  ТОЛЬКО на Apply (`SET_MODE`). Поэтому открыть vidmode безопасно — экран не «прыгает».
- **Модульность.** Один драйвер на семейство карт; легко добавлять новые.

VBE — где можно (современный BIOS с VESA 2.0/3.0 через `vesa20.sys` + V86), нативный код —
где нужно (старые карты VESA 1.x: `clgd54xx.sys`, `s3vid.sys`, `vidintel.sys`).

## 2. Формат сервиса vidmode

### 2.1. Регистрация

Драйвер — это PE-модуль ядра (`format PE DLL native 0.05`, `entry START`). При инициализации
он регистрирует сервис под своим именем:

```asm
invoke  RegService, srv_name, service_proc   ; srv_name db 'mychip',0
```

vidmode грузит драйвер функцией `68.16` по имени и общается с ним через `68.17` (IOCTL).

### 2.2. Структура IOCTL (`ioctl_t`)

`service_proc` получает (stdcall) указатель на `ioctl_t`:

| Смещение | Поле        | Назначение                          |
|---------:|-------------|-------------------------------------|
| `+0`     | handle      | (не используется драйвером)         |
| `+4`     | io_code     | код команды (1/2/3)                 |
| `+8`     | input       | указатель на вход                   |
| `+12`    | inp_size    | размер входа                        |
| `+16`    | output      | указатель на выход                  |
| `+20`    | out_size    | размер выхода                       |

### 2.3. Три команды

| Код | Имя             | Вход            | Выход                                   |
|----:|-----------------|-----------------|-----------------------------------------|
| 1   | `VBE_GET_MODES` | —               | `[dd count][count × запись 24 байта]`   |
| 2   | `VBE_SET_MODE`  | `[dd index]`    | —                                       |
| 3   | `VBE_GET_INFO`  | —               | `[dw version][dd pci_lfb]` (8 байт)     |

**`VBE_GET_INFO`**: `version` — слово, старший байт = major, младший = minor
(напр. `0x0200` = VESA 2.0). **`0` = нативный драйвер** (пикер покажет «native»).
`pci_lfb` — необязательная диагностика, пикер её не использует.

Коды `4+` драйвер может использовать под свои приватные расширения (vidmode их не
трогает). Пример — 2D-ускорение `clgd54xx` (после `SET_MODE`, координаты в пикселях
текущего режима, результат `0`/`-1`):

| Код | Имя            | Вход                                            |
|----:|----------------|--------------------------------------------------|
| 4   | `HW_FILL_RECT` | `[dd x, y, w, h, color]` (color = `0x00RRGGBB`)  |
| 5   | `HW_BLIT`      | `[dd sx, sy, dx, dy, w, h]` (копия с overlap-ом) |

### 2.4. Запись режима (24 байта, `REC_SIZE`)

| Смещение | Тип | Поле     | Описание                                                        |
|---------:|-----|----------|----------------------------------------------------------------|
| `+0`     | dd  | width    | ширина, px                                                     |
| `+4`     | dd  | height   | высота, px                                                     |
| `+8`     | dd  | bpp      | бит на пиксель (16/24/32)                                      |
| `+12`    | dw  | vbe_mode | номер VBE-режима (для нативных — `0`)                          |
| `+14`    | dw  | freq     | частота, Гц (необязательно; `0` = неизвестно)                  |
| `+16`    | dd  | phys     | физ. адрес LFB (`0` — если кадровый буфер остаётся загрузочным)|
| `+20`    | dd  | pitch    | байт на строку (обычно `width*bpp/8`)                          |

> Поле `freq` (`+14`) сейчас заполняет только `kms`, и пикер показывает Гц **только для
> него** (у остальных в `+14` мусор — там лежит старшее слово `vbe_mode`/указателя).
> Если будешь отдавать частоту из нативного драйвера — согласуй отображение с vidmode.

`GET_MODES` копирует `[count]` + массив записей в `output`. Драйвер обычно держит готовую
таблицу `mode_table` и делает `rep movsd`.

### 2.5. Применение режима (`SET_MODE`)

Вход — `[dd index]` (индекс в твоём списке режимов). Драйвер:

1. Берёт запись `mode_table[index]`.
2. **Программирует регистры чипа** (PLL/тайминги/разблокировка LFB) и включает режим.
3. Зовёт ядро, чтобы переключить весь вывод на новый кадровый буфер:

```asm
; void __stdcall SetLfbMode(width, height, bpp, pitch, lfb_phys)
push    dword [phys]
push    dword [pitch]
push    dword [bpp]
push    dword [height]
push    dword [width]
call    dword [SetLfbMode]
```

`SetLfbMode` сам выбирает блиттеры под bpp, выключает планарную конвертацию,
перенаправляет `LFB_BASE`, пересчитывает геометрию, подрезает мышь/окна, переустанавливает
курсор. Драйверу это дублировать **не нужно**.

> Драйверы Intel/EDID (`vidintel`) могут не звать `SetLfbMode`, а переиспользовать
> загрузочный LFB через `GetDisplay`/`SetScreen` — если меняют только геометрию конвейера.

### 2.6. Жизненный цикл драйвера

```asm
format PE DLL native 0.05
entry START
section '.flat' readable writable executable
include '../proc32.inc'
include '../struct.inc'
include '../macros.inc'

REASON_INIT   = 1
IOCTL_io_code = 4
IOCTL_input   = 8
IOCTL_output  = 16
VBE_GET_MODES = 1
VBE_SET_MODE  = 2
VBE_GET_INFO  = 3

proc START c uses ebx esi edi, reason:dword, cmdline:dword
        cmp     [reason], REASON_INIT
        jne     .stop
        call    DetectHardware          ; PCI/порты — детект СВОЕЙ карты
        test    eax, eax
        jz      .stop                   ; не наша карта -> не грузимся
        call    BuildModeTable          ; заполнить mode_table + mode_count
        invoke  RegService, srv_name, service_proc
        ret
.stop:
        xor     eax, eax                ; 0 -> сервис не зарегистрирован
        ret
endp

proc service_proc stdcall uses ebx esi edi, ioctl:dword
        mov     edi, [ioctl]
        mov     eax, [edi + IOCTL_io_code]
        cmp     eax, VBE_GET_MODES
        je      .get_modes
        cmp     eax, VBE_SET_MODE
        je      .set_mode
        cmp     eax, VBE_GET_INFO
        je      .get_info
        or      eax, -1                 ; неизвестная команда
        ret
        ; ... .get_modes / .set_mode / .get_info ...
endp

include '../peimport.inc'               ; импорты функций ядра
```

**Правила:**

- `START` при `REASON_INIT` детектит **своё** железо. Чужое → `xor eax,eax; ret`
  (драйвер не регистрируется, в пикере не появляется).
- **Загрузка экран не трогает.** Никаких `SetLfbMode`/смены режима в `START` — иначе при
  опросе vidmode экран «прыгнет» (этим грешил старый `vidintel`, исправлено).
- Сервис резидентен: память (MMIO и т.п.), нужную для `SET_MODE`, не освобождать.

### 2.7. Функции ядра (через `peimport.inc`)

`RegService`, `GetService`, `GetDisplay`, `SetScreen`, `SetFramebuffer`, **`SetLfbMode`**,
`PciRead32/16/8`, `MapIoMem`, `FreeKernelSpace`, **`V86start`/`V86machine`** (вызов видео-BIOS
`int 10h` — так делает `vesa20`), `SysMsgBoardStr` (лог) и др.

### 2.8. Регистрация в пикере и сборка

Добавь имя сервиса в `name_table` приложения `programs/system/vidmode/vidmode.asm` и при
необходимости подними `MAX_DRV`. Драйверы лежат в `drivers/video/` и собираются в `*.sys`
(`Tupfile.lua` глобит `*.asm`); fasm: `fasm mychip.asm mychip.sys`.

## 3. Формат сервиса KMS («DISPLAY»)

Портированные Сержем DRM-драйверы (`i915`/`radeon`/`vmwgfx`, каталог `drivers/video/drm/`)
используют **другой** протокол. Это полноценные порты ядра Linux (GEM/TTM/atomic
modesetting), собираются `kos32-gcc` + DDK в PE-`.dll`.

- Регистрируются под **единым именем `"DISPLAY"`**: `RegService("DISPLAY", display_handler)`.
  Это **синглтон** — активен только один такой сервис.
- Сами **владеют кадровым буфером** (GEM/TTM) и делают полный KMS (DPLL, CRTC, коннекторы,
  EDID, DP/HDMI). Кадровый буфер ядру отдают через `GetDisplay`.

### 3.1. Коды (тот же `ioctl_t`, 68.17)

| Код | Имя              | Семантика                                                  |
|----:|------------------|------------------------------------------------------------|
| 0   | `SRV_GETVERSION` | `output(dd)` = версия (`compat<<16 | current`)             |
| 1   | `SRV_ENUM_MODES` | **двухпроходный**, см. ниже                                |
| 2   | `SRV_SET_MODE`   | `input` = `videomode_t` (16 байт)                          |
| 3   | `SRV_GET_CAPS`   | возможности драйвера                                        |
| 4   | `SRV_CMDLINE`    | командная строка                                            |
| 10+ | …                | GEM/surface-операции (3D/блиттер для Gallium)              |

### 3.2. `videomode_t` (16 байт)

```c
typedef struct { int width, height, bpp, freq; } videomode_t;   // bpp = 32
```

### 3.3. `SRV_ENUM_MODES` (два прохода)

Единицы — **штуки режимов** (не пиксели):

1. **Сколько режимов:** `output → &count`, `count = 0` на входе → драйвер кладёт в `count`
   общее число режимов коннектора.
2. **Заполнить:** `input → videomode_t[]`, `output → &count` (на входе = желаемое число) →
   драйвер заполняет `min(count, total)` записей и кладёт в `count` фактическое число.

### 3.4. `SRV_SET_MODE`

`input` = `videomode_t{w,h,bpp,freq}`. Драйвер ищет режим в списке коннектора (точное
совпадение `w,h,freq`, при `strict=false` — по `w,h`) и переключает KMS-конвейер.

## 4. Почему мост, а не драйверы в архитектуре KMS

Когда у KolibriOS уже есть KMS-драйверы (`i915`/`radeon`/`vmwgfx`), возникает соблазн писать
все новые драйверы «как KMS». Мы пошли другим путём — нативные `.sys` в формате vidmode + один
**мост** `kms.sys`. Почему:

- **KMS-драйверы огромны и тяжелы в сборке.** Это апстримные порты Linux DRM (десятки файлов,
  сотни тысяч строк C), требуют `kos32-gcc` + DDK. Для карты VESA 1.x это стрельба из пушки
  по воробьям: там нужно лишь запрограммировать пару регистров чипа, разблокировать LFB и
  позвать `SetLfbMode` — **~сотни строк FASM**, только `fasm`, без всякого тулчейна.
- **KMS не ускоряет 2D-десктоп KolibriOS.** Рабочий стол рисуется софтовыми блиттерами ядра
  в LFB независимо от того, KMS поставил режим или нативный драйвер. Полный KMS-стек ради
  старой карты не даёт выигрыша в отрисовке (см. отдельный разбор про блиттер Cirrus).
- **Мост не форкает апстрим.** DRM-порты остаются бит-в-бит, как у Сержа; весь
  KolibriOS-специфичный клей — в одном маленьком FASM-файле. Это **шов на границе
  «свой код / вендорный код»** — правильное место для шва.
- **`"DISPLAY"` — синглтон, а vidmode — мульти-драйвер-по-имени.** Нельзя зарегистрировать
  несколько `"DISPLAY"`. Мост выставляет единственный `DISPLAY` как одну строку `kms` в
  общем пикере рядом с нативными драйверами.

Итог: переписывать KMS-драйвер под формат vidmode бессмысленно (выбросишь 99% и потеряешь
3D/GEM), а реализовать формат vidmode «поверх» KMS дёшево — это и есть мост.

## 5. Формат моста `kms.sys`

`drivers/video/kms.asm` — тонкий транслятор (~190 строк FASM). **Снаружи** это обычный
сервис vidmode (`kms`), **внутри** он вызывает сервис `"DISPLAY"` напрямую.

### 5.1. Как мост зовёт чужой сервис

Загруженный драйвер может вызвать другой сервис прямо в ядре, без `68.17`:

```asm
SRV_PROC = 0x28                         ; смещение user-mode хендлера в struct SRV

invoke  GetService, sz_display          ; sz_display db 'DISPLAY',0 -> eax = SRV*
test    eax, eax
jz      .nodisp                         ; нет KMS -> НЕ регистрируемся (не светимся в пикере)
mov     ecx, [eax + SRV_PROC]           ; ecx = display_handler
mov     [disp_proc], ecx
; вызов: положить ioctl_t в d_ioctl и
push    d_ioctl
call    dword [disp_proc]               ; stdcall: callee чистит аргумент
```

### 5.2. Трансляция протоколов

- **`VBE_GET_MODES`** → двухпроходный `SRV_ENUM_MODES` (сначала count, затем заполнить
  `videomode_t[]`); репак `videomode_t{w,h,bpp,freq}` → 24-байтная запись: `freq` кладётся
  в `+14`, `vbe_mode/phys/pitch = 0` (кадровым буфером владеет DRM). Список `videomode_t`
  **кэшируется** — он нужен для `SET_MODE` по индексу.
- **`VBE_SET_MODE`** (индекс) → берём кэшированный `videomode_t[index]` → `SRV_SET_MODE`.
- **`VBE_GET_INFO`** → версия `0` («native»), lfb `0`.

Мост **не трогает железо и `SetLfbMode`** — это делает DRM-драйвер. Если `"DISPLAY"` не
зарегистрирован (KMS не загружен), мост не делает `RegService`, и пикер его не показывает.

## 6. Какой формат выбирать

| Ситуация | Формат | Почему |
|----------|--------|--------|
| Конкретная **старая карта** (VESA 1.x), прямое программирование регистров, линейный буфер, софт-десктоп | **Нативный `.sys` (формат vidmode)** | Легко (сотни строк FASM, только fasm), сосуществует с другими, переиспользует ядро (`SetLfbMode`/`V86`), попадает в пикер. **Дефолт для старого железа.** |
| **Современный GPU**: нужны коннекторы/EDID/DP/HDMI, GEM/TTM, 3D (Gallium); или уже есть Linux DRM-драйвер для порта | **KMS/«DISPLAY»** | Это полный бэкенд дисплея. Тяжело (C + kos32-gcc + DDK), но даёт весь стек. |
| KMS/«DISPLAY»-драйвер **уже существует**, надо показать его в пикере рядом с нативными | **Мост (как `kms.sys`)** | Не переписывай — сделай тонкий транслятор протоколов. ~190 строк, собирается `fasm`. |

**Эвристика:**

- Можешь поставить режим, записав регистры чипа и выставив линейный буфер? → нативный
  драйвер формата vidmode.
- Нужен полноценный KMS (или уже есть готовый порт)? → формат «DISPLAY».
- «DISPLAY» уже есть, нужен лишь пункт в пикере? → мост.

## 7. Чек-лист нового нативного драйвера

1. `format PE DLL native 0.05`, `entry START`, инклюды `proc32/struct/macros`, в конце
   `peimport.inc`.
2. `START`: при `REASON_INIT` детект **своей** карты (PCI/порты). Чужая → `xor eax,eax; ret`.
   **Экран не трогать.**
3. Построить `mode_table` (24-байтные записи) и `mode_count`.
4. `RegService, srv_name, service_proc`.
5. `service_proc`: реализовать `GET_MODES` / `SET_MODE` / `GET_INFO` (см. §2.3–2.5).
6. `SET_MODE`: запрограммировать чип + `SetLfbMode(w,h,bpp,pitch,phys)`.
7. `GET_INFO`: версия `0` для нативного драйвера.
8. Добавить имя в `name_table` приложения `vidmode` (и при нужде `MAX_DRV`).
9. Собрать: `fasm mychip.asm mychip.sys`, положить рядом с прочими `*.sys`.

---

*См. примеры: `vesa20.asm` (VBE через V86), `s3vid.asm` (порт Linux s3fb), `clgd54xx.asm`
(Cirrus, нативные 16/24-бит), `vidintel.asm` (Intel, режим по EDID), `kms.asm` (мост).*
